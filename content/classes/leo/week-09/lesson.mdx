## Big Idea — Instructions Are Just Numbers

You have an ALU that does math. You have registers that hold numbers. But how does the CPU know *what to do*? Who tells it "add register 1 and register 2"?

The answer is the most beautiful idea in all of computing: **an instruction is just a number.**

A program is a list of these numbers. The CPU reads them one at a time and does what each one says. Reading the next number is called **fetch**. Figuring out what it means is **decode**. Doing it is **execute**. Over and over, billions of times a second. That loop is the heartbeat of every computer ever built.

---

## Packing an Instruction Into One Number

Our little CPU uses an **8-bit instruction**. We split those 8 bits into four 2-bit fields:

```
   bits:  7 6   5 4   3 2   1 0
         [ OP ][DEST][SRCA][SRCB]
```

| Field | Bits | Meaning |
|-------|------|---------|
| **OP** | 7-6 | which operation (0=ADD, 1=SUB, 2=AND, 3=OR) |
| **DEST** | 5-4 | which register to store the answer in |
| **SRCA** | 3-2 | first input register |
| **SRCB** | 1-0 | second input register |

Each field is 2 bits, so each can name one of 4 things — 4 operations, 4 registers. Neat and tidy.

### Example: "ADD register 1 and register 2, store in register 0"

- OP = 0 (ADD), DEST = 0, SRCA = 1, SRCB = 2
- In bits: `00 00 01 10`
- As one number: `00000110` = **6**

So the single number **6** means "r0 = r1 + r2". The whole instruction, packed into one byte!

---

## Decoding — Unpacking the Fields

To run an instruction, the CPU first **decodes** it — pulls each field back out. This is exactly the shift-and-mask trick from Week 2's `get_bit`, just grabbing 2 bits at a time instead of 1.

To grab a 2-bit field: **shift it down** to the bottom, then **mask with `& 3`** (because `3` is `11` in binary — it keeps the lowest 2 bits).

```python
def get_opcode(instr):
    return (instr >> 6) & 3    # bits 7-6 -> shift down 6

def get_dest(instr):
    return (instr >> 4) & 3    # bits 5-4 -> shift down 4

def get_srcA(instr):
    return (instr >> 2) & 3    # bits 3-2 -> shift down 2

def get_srcB(instr):
    return instr & 3           # bits 1-0 -> already at the bottom
```

### Trace: decoding `6`

- `6` is `00000110`
- `get_opcode(6)` = `(6 >> 6) & 3` = `0`
- `get_dest(6)` = `(6 >> 4) & 3` = `0`
- `get_srcA(6)` = `(6 >> 2) & 3` = `1`
- `get_srcB(6)` = `6 & 3` = `2`

Decoded: OP=0 (ADD), DEST=0, SRCA=1, SRCB=2 → "r0 = r1 + r2".

## Executing — Doing What It Says

Now the payoff. `execute` takes the current registers (a list of 4 numbers) and one instruction. It decodes the instruction, reads the two source registers, runs the ALU, and stores the answer in the destination register.

```python
def execute(regs, instr):
    op   = get_opcode(instr)
    dest = get_dest(instr)
    a    = get_srcA(instr)
    b    = get_srcB(instr)

    result = alu(regs[a], regs[b], op)   # your ALU from Week 7!

    new_regs = list(regs)    # make a copy so we don't disturb the original
    new_regs[dest] = result  # change just the destination slot
    return new_regs
```

Notice we **copy** the register list and change only one slot, then return the new list. The CPU's registers march forward one step at a time — each instruction produces a fresh snapshot.

### Trace: `execute([0, 5, 3, 0], 6)`

- Decode `6`: OP=ADD, DEST=0, SRCA=1, SRCB=2
- Read sources: `regs[1]` = 5, `regs[2]` = 3
- ALU: `alu(5, 3, 0)` = 8
- Store in r0: registers become `[8, 5, 3, 0]` ✓

The CPU just ran an instruction!

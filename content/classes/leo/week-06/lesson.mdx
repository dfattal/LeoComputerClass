## Big Idea — From Sticky Note to Filing Cabinet

Last week, you built a D flip-flop — a circuit that remembers one bit. That's like having a single sticky note that holds a 0 or a 1.

But one bit isn't very useful on its own. You can't store a real number with one bit — you need several bits working together. And a CPU doesn't just need one number stored — it needs a whole set of numbers it can grab quickly.

So this week, you'll go from **one sticky note** → **a row of sticky notes** (a register) → **a filing cabinet full of rows** (a register file).

By the end, you'll have built the storage system that sits right inside the processor — the fastest memory a computer has.

## The 1-Bit Register

### Simplifying the Flip-Flop

Last week's D flip-flop had four inputs: d, clk, clk_prev, q. That's a lot to keep track of. In practice, every flip-flop in a CPU shares the same clock — they all tick together. So from now on, we'll assume the clock is ticking and focus on just the **load** signal:

- **load = 1:** Store the new value d
- **load = 0:** Keep the old value q

This is called a **1-bit register** (or "register bit"). It's the same MUX trick from the D latch:

```python
def reg_bit(d, load, q):
    return mux(q, d, load)
```

Think of `load` like a valve:
- Valve open (load=1): new data flows in
- Valve closed (load=0): the pipe is sealed, nothing changes

## The 4-Bit Register

### A Row of Sticky Notes

One sticky note = one bit. Want to store a 4-bit number? Use **four sticky notes side by side** — four `reg_bit` circuits, all sharing the same `load` signal.

When `load=1`, all four bits update at once with new data. When `load=0`, all four bits hold their values. It's like a team — they all load together or all hold together.

### Building It

Your `reg4` function takes a 4-bit input `d`, a `load` signal, and the current 4-bit stored value `q`. It returns the new 4-bit value.

Since `d` and `q` are regular numbers (0–15), you need to break them into individual bits, run each through `reg_bit`, then reassemble the result. This is the same pattern you used for `add4` in Week 4!

```python
def reg4(d, load, q):
    # Break d and q into individual bits
    d0, d1, d2, d3 = get_bit(d, 0), get_bit(d, 1), get_bit(d, 2), get_bit(d, 3)
    q0, q1, q2, q3 = get_bit(q, 0), get_bit(q, 1), get_bit(q, 2), get_bit(q, 3)

    # Run each bit through reg_bit
    new0 = reg_bit(d0, load, q0)
    new1 = reg_bit(d1, load, q1)
    new2 = reg_bit(d2, load, q2)
    new3 = reg_bit(d3, load, q3)

    # Reassemble into a number
    return (new3 << 3) | (new2 << 2) | (new1 << 1) | new0
```

### Trace: `reg4(9, 1, 5)`

- d=9 (binary 1001), load=1, q=5 (binary 0101)
- Since load=1, every reg_bit picks d over q
- new3=1, new2=0, new1=0, new0=1
- Result: 1001 = **9** (loaded the new value!)

### Trace: `reg4(9, 0, 5)`

- d=9, load=0, q=5
- Since load=0, every reg_bit picks q over d
- new3=0, new2=1, new1=0, new0=1
- Result: 0101 = **5** (held the old value!)

## The 4-Way MUX — Picking From Four Options

### Why We Need It

You built a 2-way MUX in Week 2 — it picks between **two** values using one selector bit. But a register file has **four** registers. To read one of them, you need to pick between **four** values.

### The Post Office Analogy

Imagine a post office with 4 mailboxes, numbered 0 through 3. You walk up and say a number (the **address**), and the clerk hands you that mailbox's contents.

A 4-way MUX does exactly this. It takes four input values and a 2-bit address, and outputs the value at that address.

### Building MUX4 from Three MUX2s

Here's the trick: you can build a 4-way MUX using three of the 2-way MUX you already have! Think of it as a tournament bracket:

```
a ──┐
     MUX (s0) → winner1 ──┐
b ──┘                       MUX (s1) → output
c ──┐                      │
     MUX (s0) → winner2 ──┘
d ──┘
```

- **Round 1:** s0 picks between a/b, and separately between c/d
- **Final round:** s1 picks between the two round-1 winners

The address bits are s1 and s0, where:
- s1=0, s0=0 → picks a (address 0)
- s1=0, s0=1 → picks b (address 1)
- s1=1, s0=0 → picks c (address 2)
- s1=1, s0=1 → picks d (address 3)

### Walking Through It

Let's say a=10, b=20, c=30, d=40, and the address is 2 (s1=1, s0=0):

1. `mux(a, b, s0)` = `mux(10, 20, 0)` → **10** (s0=0, pick a)
2. `mux(c, d, s0)` = `mux(30, 40, 0)` → **30** (s0=0, pick c)
3. `mux(10, 30, s1)` = `mux(10, 30, 1)` → **30** (s1=1, pick second winner)

Output: **30**, which is c — register 2. Correct!

## Python Lists — A New Tool

Before we build the register file, you need one new Python concept: **lists**.

### What's a List?

A list is like a row of numbered boxes, each holding a value. You create one with square brackets:

```python
regs = [10, 20, 30, 40]
```

This creates 4 boxes:
- Box 0 holds **10**
- Box 1 holds **20**
- Box 2 holds **30**
- Box 3 holds **40**

### Reading from a List

Use square brackets with the box number (called the **index**):

```python
regs[0]   # → 10
regs[2]   # → 30
```

Indices start at 0, not 1! This is just like bit numbering — the rightmost bit is bit 0.

### Why Lists Matter for Registers

A register file is a collection of registers, each holding a value. A list is the perfect way to represent this — each slot in the list is one register.

```python
regs = [0, 0, 0, 0]   # four registers, all starting at 0
```

## The Register File — A Tiny Memory Bank

### What It Does

A register file is a small, super-fast memory bank inside the CPU. Ours has:
- **4 registers** (numbered 0 to 3), each storing a 4-bit value
- **Read**: give an address (0–3), get back that register's value

Think of it like a tiny filing cabinet with 4 drawers. The address tells you which drawer to open.

### Reading from the Register File

To read, you use your MUX4! The four register values go in as inputs, and the address selects which one comes out.

```python
def reg_file_read(regs, addr):
    s0 = get_bit(addr, 0)
    s1 = get_bit(addr, 1)
    return mux4(regs[0], regs[1], regs[2], regs[3], s1, s0)
```

That's it! The MUX4 does all the work — it picks the register at the given address. Notice that reading from a register file doesn't use any memory circuits — it's entirely MUXes (combinational logic). The memory is in the registers themselves; reading just selects which one to show.

### Why Registers Are Special

Your computer has gigabytes of RAM, but only a handful of registers. Why bother with such a tiny memory? **Speed.**

RAM is far away from the CPU — data has to travel through wires, wait for memory chips, and come back. That takes hundreds of clock ticks. Registers sit right inside the CPU, so reading one takes just **one tick**.

When the CPU is doing math, it loads numbers into registers, does the calculation, and stores the result back. Registers are the CPU's scratch pad — small, but lightning fast.

## What You Just Built

You went from a single-bit sticky note to a complete memory bank! Here's the stack:

- **reg_bit** — stores 1 bit (a MUX that picks between "hold" and "load")
- **reg4** — stores a 4-bit number (four reg_bits working together)
- **mux4** — selects one of four values (the address decoder for reading)
- **reg_file_read** — reads any register by address (the CPU's fast scratch pad)

This is real. Every CPU on the planet has a register file just like this — a small set of named storage bins that the processor reads and writes thousands of times per microsecond.
